# 백업 성공 기준을 “파일 생성”에서 “복원 가능성”으로 확장하기

## 개요

이번 작업에서는 DB FleetOps의 백업 기능을 한 단계 확장했습니다.

기존에는 MySQL 백업 작업이 성공했는지를 다음 기준으로 판단했습니다.

```text
mysqldump 실행
  ↓
.sql 파일 생성
  ↓
파일 크기 확인
  ↓
checksum 생성
  ↓
Backup Job 성공
```

이 방식은 백업 파일이 생성됐다는 사실은 확인할 수 있지만, 실제 장애 상황에서 그 파일로 DB를 복원할 수 있는지는 보장하지 못합니다.

그래서 이번 작업에서는 백업 성공 기준을 다음과 같이 바꿨습니다.

```text
mysqldump 실행
  ↓
.sql 파일 생성
  ↓
임시 DB 생성
  ↓
백업 파일 Restore
  ↓
테이블 존재 여부 확인
  ↓
Row Count 확인
  ↓
복원 검증 결과 저장
  ↓
Backup Job 성공 또는 실패 처리
```

핵심은 백업 성공을 파일 생성이 아니라 복원 가능성 기준으로 판단하도록 만든 것입니다.

---

## 문제의식

백업 기능에서 가장 위험한 상황은 백업 파일이 존재하지만 실제로 복원되지 않는 경우입니다.

예를 들어 다음과 같은 문제가 있을 수 있습니다.

- 백업 파일은 생성됐지만 SQL이 깨져 있는 경우
- 특정 테이블이 누락된 경우
- dump 옵션 문제로 다른 DB에 복원할 수 없는 경우
- 파일 크기와 checksum은 정상인데 실제 restore 단계에서 실패하는 경우
- 백업 성공 이력은 있지만 복원 검증 이력이 없어 장애 대응 근거가 부족한 경우

운영 관점에서 백업은 “파일을 만들었다”보다 “필요할 때 복원할 수 있다”가 더 중요합니다.

그래서 이번 작업에서는 백업 Job의 성공 기준을 더 보수적으로 잡았습니다.

---

## 전체 처리 흐름

이번 작업 이후 백업 Job의 흐름은 다음과 같습니다.

```text
OperationJob: BACKUP
  ↓
OperationTask: MYSQL_LOGICAL_BACKUP
  ↓
mysqldump 실행
  ↓
백업 파일 생성
  ↓
백업 파일 기본 검증
  ↓
OperationTask: MYSQL_RESTORE_VERIFY 생성
  ↓
임시 DB 생성
  ↓
백업 파일 restore
  ↓
SHOW TABLES
  ↓
expectedTables 존재 여부 확인
  ↓
SELECT COUNT(*) FROM table
  ↓
임시 DB 삭제
  ↓
BackupRestoreVerification 저장
  ↓
OperationJob SUCCEEDED 또는 FAILED 처리
```

기존에는 `MYSQL_LOGICAL_BACKUP` Task가 성공하면 바로 `OperationJob`을 성공 처리했습니다.

이번 작업 이후에는 `MYSQL_LOGICAL_BACKUP`이 성공해도 바로 Job을 성공 처리하지 않습니다.

```text
기존 흐름

MYSQL_LOGICAL_BACKUP 성공
  ↓
OperationJob SUCCEEDED
```

변경된 흐름은 다음과 같습니다.

```text
변경 후 흐름

MYSQL_LOGICAL_BACKUP 성공
  ↓
MYSQL_RESTORE_VERIFY Task 생성
  ↓
OperationJob RUNNING 유지
  ↓
MYSQL_RESTORE_VERIFY 성공
  ↓
OperationJob SUCCEEDED
```

즉, 백업 파일 생성은 중간 단계이고, 실제 복원 검증까지 통과해야 Backup Job이 성공합니다.

## Go Agent 쪽 구현

이번 작업에서 Go Agent에는 복원 실행과 검증 로직을 추가했습니다.

역할은 크게 세 가지로 나눴습니다.

```text
MySQLRestoreRunner
  - 임시 DB 생성
  - mysql client로 백업 파일 restore
  - 임시 DB cleanup

MySQLRestoreVerifier
  - SHOW TABLES 실행
  - expectedTables 존재 여부 확인
  - SELECT COUNT(*) 실행

MySQLRestoreVerifyHandler
  - Task Payload 파싱
  - Runner 실행
  - Verifier 실행
  - Cleanup 실행
  - 결과 JSON 반환
```

## MySQLRestoreRunner 설계

`MySQLRestoreRunner`는 실제 restore 명령 실행만 담당합니다.

처리 흐름은 다음과 같습니다.

```text
validate request
  ↓
defaults-extra-file 생성
  ↓
CREATE DATABASE temporary_db
  ↓
mysql temporary_db < backup.sql
  ↓
필요 시 DROP DATABASE temporary_db
```

복원 실행 시 shell redirection을 직접 사용하지 않았습니다.

잘못된 방식은 다음과 같습니다.

```bash
mysql restore_db < backup.sql
```

대신 Go 코드에서는 backup file을 열어서 `command.Stdin`으로 연결했습니다.

```go
command.Stdin = file
```

이렇게 한 이유는 다음과 같습니다.

- shell 의존성을 줄일 수 있음
- command argument와 stdin을 분리할 수 있음
- 테스트에서 fake command runner로 검증하기 쉬움
- 비밀번호를 command argument에 직접 노출하지 않을 수 있음

MySQL 접속 정보는 `--defaults-extra-file`을 사용했습니다.

```text
[client]
host=127.0.0.1
port=3306
user=restore_user
password=secret
```

이 파일은 임시 파일로 만들고, 실행 후 삭제합니다.

## Cleanup 전용 메서드 분리

처음에는 `Run()` 메서드의 `Cleanup=true` 옵션만으로 cleanup까지 처리하려고 했습니다.

하지만 restore 검증 흐름에서는 문제가 있었습니다.

복원 검증을 하려면 restore 직후 임시 DB가 남아 있어야 합니다.

```text
restore 실행
  ↓
임시 DB 유지
  ↓
SHOW TABLES
  ↓
COUNT 조회
  ↓
cleanup
```

따라서 `Run(cleanup=true)`로 한 번에 처리하면 검증 전에 임시 DB가 삭제될 수 있습니다.

그래서 cleanup 전용 메서드를 분리했습니다.

```go
func (r *MySQLRestoreRunner) Cleanup(
    ctx context.Context,
    request MySQLRestoreCleanupRequest,
) error
```

최종 흐름은 다음과 같습니다.

```text
Run(cleanup=false)
  ↓
Verifier.Verify()
  ↓
Cleanup()
```

이렇게 분리하면서 Runner의 책임도 더 명확해졌습니다.

## 임시 DB 이름 검증

복원 검증에서는 임시 DB를 생성합니다.

예시는 다음과 같습니다.

```text
restore_verify_orders_100_20260708_140000
```

DB 이름은 SQL identifier이므로 일반적인 값 바인딩 대상이 아닙니다.

따라서 임시 DB 이름에는 안전한 문자만 허용했습니다.

```text
a-z
A-Z
0-9
_
```

허용하지 않는 예시는 다음과 같습니다.

```text
restore-verify-orders
orders;drop
orders/test
orders prod
```

이 정책은 다소 보수적이지만, 운영용 자동화 도구에서는 안전한 기본값이 더 중요하다고 판단했습니다.

## MySQLRestoreVerifier 설계

`MySQLRestoreVerifier`는 restore된 임시 DB를 검증합니다.

검증 기준은 다음과 같습니다.

```text
SHOW TABLES 성공
AND
복원된 테이블이 1개 이상 존재
AND
expectedTables가 있으면 모두 존재
AND
verifyRowCount=true이면 Row Count 조회 성공
```

Row Count가 0인 것은 실패로 보지 않았습니다.

빈 테이블도 정상적인 테이블일 수 있기 때문입니다.

예를 들어 기준 테이블은 존재하지만 아직 데이터가 없는 경우도 있습니다.

```json
{
  "tableName": "audit_log",
  "existsInRestoredDb": true,
  "rowCount": 0,
  "status": "VERIFIED"
}
```

따라서 Row Count는 데이터 검증 정보로 저장하되, 값이 0이라는 이유만으로 실패 처리하지 않았습니다.

## Task Type 추가

새로운 Task Type을 추가했습니다.

```java
MYSQL_RESTORE_VERIFY
```

기존 백업 Task와 새 복원 검증 Task의 관계는 다음과 같습니다.

```text
MYSQL_LOGICAL_BACKUP
  ↓
MYSQL_RESTORE_VERIFY
```

Control Plane과 Go Agent 양쪽에서 동일한 문자열을 사용하도록 맞췄습니다.

```go
const TaskTypeMySQLRestoreVerify = "MYSQL_RESTORE_VERIFY"
```

```java
MYSQL_RESTORE_VERIFY
```

Task Type 문자열이 어긋나면 Agent Dispatcher가 Handler를 찾지 못하기 때문에, 양쪽 이름을 명확히 맞추는 것이 중요했습니다.

---

## Handler 연결

Go Agent의 Dispatcher는 Handler 목록을 순회하면서 지원 가능한 Task Type을 찾는 구조입니다.

```go
for _, handler := range d.handlers {
    if handler.Supports(task.TaskType) {
        return handler.Handle(ctx, task)
    }
}
```

따라서 기존 Dispatcher 구조는 수정하지 않고, main에서 Handler만 추가했습니다.

```go
dispatcher :=
    task.NewDispatcher(
        []task.Handler{
            task.NewLinuxStatusHandler(linuxStatusCollector),
            task.NewMySQLBackupHandler("/tmp/db-fleetops-backups"),
            task.NewMySQLRestoreVerifyHandler(),
        },
    )
```

이 구조의 장점은 Task가 늘어나도 Dispatcher 자체는 변경하지 않아도 된다는 점입니다.

새로운 Task를 추가할 때는 Handler를 구현하고 등록만 하면 됩니다.

## Control Plane 쪽 구현

Control Plane에서는 백업 Task 완료 후 복원 검증 Task를 생성하도록 변경했습니다.

기존 `completeTask()` 흐름은 단순했습니다.

```text
Task 완료
  ↓
Linked OperationJob 조회
  ↓
OperationJob 성공 처리
```

이번 작업 이후에는 Task Type에 따라 분기합니다.

```text
COLLECT_LINUX_STATUS
  ↓
Metric 저장

MYSQL_LOGICAL_BACKUP
  ↓
verifyAfterBackup 확인
  ↓
true이면 MYSQL_RESTORE_VERIFY Task 생성
  ↓
false이면 OperationJob 성공 처리

MYSQL_RESTORE_VERIFY
  ↓
복원 검증 결과 저장
  ↓
OperationJob 성공 또는 실패 처리
```

## verifyAfterBackup 옵션

모든 백업에서 항상 복원 검증을 수행하는 것이 이상적이지만, 개발 환경이나 테스트 환경에서는 복원 검증용 MySQL 환경이 준비되지 않을 수 있습니다.

그래서 `verifyAfterBackup` 옵션을 두었습니다.

```json
{
  "verifyAfterBackup": false
}
```

이 값이 `false`이면 기존처럼 백업 파일 생성 결과만으로 Job을 성공 처리합니다.

```text
MYSQL_LOGICAL_BACKUP 완료
  ↓
verifyAfterBackup=false
  ↓
OperationJob SUCCEEDED
```

반대로 `true`이면 복원 검증 Task를 생성합니다.

```text
MYSQL_LOGICAL_BACKUP 완료
  ↓
verifyAfterBackup=true
  ↓
MYSQL_RESTORE_VERIFY Task 생성
  ↓
OperationJob RUNNING 유지
```

운영 기본값은 `true`가 맞지만, 현재 일부 기존 흐름과 테스트 호환성을 위해 백업 Task 생성 메서드에서는 일단 `false`로 두었습니다.

이후 DB Inventory와 Credential 정보가 백업 Task Payload에 완전히 연결되면 기본값을 `true`로 전환할 수 있습니다.

## RestoreVerifyTaskPayloadFactory

백업 Task 완료 후 복원 검증 Task를 만들기 위해 별도 Factory를 두었습니다.

이 Factory는 두 가지 JSON을 조합합니다.

```text
backup task parametersJson
backup task resultPayloadJson
```

백업 Task parametersJson에는 접속 정보와 검증 옵션이 들어갑니다.

```json
{
  "operationJobId": 100,
  "databaseId": 1,
  "databaseName": "orders",
  "host": "127.0.0.1",
  "port": 3306,
  "username": "backup_user",
  "password": "secret",
  "backupType": "LOGICAL",
  "compression": false,
  "verifyAfterBackup": true,
  "expectedTables": ["orders", "order_items"],
  "verifyRowCount": true,
  "cleanup": true
}
```

백업 Task resultPayloadJson에는 backupFile 정보가 들어갑니다.

```json
{
  "status": "VERIFIED",
  "backupFile": "/tmp/db-fleetops-backups/orders.sql",
  "fileSizeBytes": 12345,
  "checksumSha256": "abc123",
  "createdAt": "2026-07-08T14:00:00+09:00",
  "message": "backup artifact verified"
}
```

Factory는 이 둘을 조합해 복원 검증 Task Payload를 만듭니다.

```json
{
  "operationJobId": 100,
  "databaseId": 1,
  "backupTaskId": 200,
  "sourceDatabaseName": "orders",
  "backupFile": "/tmp/db-fleetops-backups/orders.sql",
  "host": "127.0.0.1",
  "port": 3306,
  "username": "backup_user",
  "password": "secret",
  "temporaryDatabaseName": "restore_verify_orders_100_20260708_140000",
  "expectedTables": ["orders", "order_items"],
  "verifyRowCount": true,
  "cleanup": true
}
```

Factory를 분리한 이유는 `OperationTaskService`가 JSON 조립 세부사항까지 알지 않도록 하기 위해서입니다.

## 복원 검증 결과 저장

복원 검증 결과는 별도 도메인으로 저장했습니다.

주요 Entity는 두 개입니다.

```text
BackupRestoreVerification
BackupRestoreVerificationItem
```

`BackupRestoreVerification`은 복원 검증 1회 실행의 요약 정보입니다.

```text
operationJobId
backupTaskId
restoreVerifyTaskId
databaseId
sourceDatabaseName
backupFile
temporaryDatabaseName
status
restoredTableCount
checkedTableCount
totalRowCount
errorCode
errorMessage
startedAt
completedAt
createdAt
```

`BackupRestoreVerificationItem`은 테이블 단위 검증 결과입니다.

```text
verificationId
tableName
existsInRestoredDb
rowCount
status
message
createdAt
```

이렇게 분리한 이유는 복원 검증 전체 결과와 테이블별 세부 결과의 성격이 다르기 때문입니다.

전체 결과는 Job 성공/실패 판단에 사용하고, Item 결과는 장애 원인 분석에 사용합니다.

## 상태 설계

복원 검증 전체 상태는 다음과 같이 정의했습니다.

```text
REQUESTED
RUNNING
VERIFIED
FAILED
CLEANUP_FAILED
```

각 상태의 의미는 다음과 같습니다.

| 상태 | 의미 |
|---|---|
| REQUESTED | 복원 검증 요청이 생성된 상태 |
| RUNNING | 복원 검증이 진행 중인 상태 |
| VERIFIED | 복원 및 최소 검증이 성공한 상태 |
| FAILED | 복원 또는 검증이 실패한 상태 |
| CLEANUP_FAILED | 검증은 수행됐지만 임시 DB 삭제에 실패한 상태 |

테이블 단위 상태는 다음과 같습니다.

```text
VERIFIED
MISSING
COUNT_FAILED
SKIPPED
```

| 상태 | 의미 |
|---|---|
| VERIFIED | 테이블 존재 확인 및 Row Count 검증 성공 |
| MISSING | 기대한 테이블이 복원 DB에 없음 |
| COUNT_FAILED | 테이블은 있지만 Row Count 조회 실패 |
| SKIPPED | Row Count 검증 생략 |

---

## Job 성공/실패 처리 기준

복원 검증 결과에 따른 Job 처리 기준은 다음과 같이 잡았습니다.

| Restore Verification 결과 | OperationJob 결과 | 이유 |
|---|---|---|
| VERIFIED | SUCCEEDED | 백업 파일이 실제로 복원 가능하다고 판단 |
| FAILED | FAILED | 복원 또는 검증 실패 |
| CLEANUP_FAILED | FAILED | 임시 DB가 남을 수 있어 운영상 불완전 종료 |
| verifyAfterBackup=false | SUCCEEDED | 복원 검증을 생략하고 파일 백업 결과만 반영 |

특히 `CLEANUP_FAILED`는 고민이 있었습니다.

복원 검증 자체는 성공했을 수 있지만, 임시 DB 삭제에 실패했다면 운영 환경에 찌꺼기 DB가 남을 수 있습니다.

따라서 Job은 보수적으로 실패 처리했습니다.

## 조회 API 추가

복원 검증 결과를 확인할 수 있도록 조회 API도 추가했습니다.

추가한 API는 세 가지입니다.

```http
GET /api/v1/jobs/{jobId}/restore-verification
```

특정 백업 Job의 복원 검증 결과를 조회합니다.

```http
GET /api/v1/databases/{databaseId}/restore-verifications/latest
```

특정 DB의 최신 복원 검증 결과를 조회합니다.

```http
GET /api/v1/restore-verifications/{verificationId}
```

복원 검증 ID 기준으로 상세 결과를 조회합니다.

운영 화면에서는 다음과 같이 사용할 수 있습니다.

```text
Job 상세 화면
  → Job 기준 복원 검증 결과 조회

DB 상세 화면
  → DB 기준 최신 복원 검증 결과 조회

복원 검증 상세 화면
  → verificationId 기준 상세 조회
```

## 테스트 보강

이번 작업에서는 테스트도 함께 보강했습니다.

주요 테스트 흐름은 다음과 같습니다.

```text
verifyAfterBackup=false
  MYSQL_LOGICAL_BACKUP 완료
  → OperationJob SUCCEEDED

verifyAfterBackup=true
  MYSQL_LOGICAL_BACKUP 완료
  → MYSQL_RESTORE_VERIFY Task 생성
  → OperationJob RUNNING 유지

MYSQL_RESTORE_VERIFY VERIFIED
  → BackupRestoreVerification 저장
  → OperationJob SUCCEEDED

MYSQL_RESTORE_VERIFY FAILED
  → BackupRestoreVerification 저장
  → OperationJob FAILED

MYSQL_RESTORE_VERIFY CLEANUP_FAILED
  → BackupRestoreVerification 저장
  → OperationJob FAILED
```

Go Agent 쪽에서는 실제 MySQL에 의존하지 않도록 fake command runner와 sqlmock을 사용했습니다.

Runner 테스트에서는 다음을 검증했습니다.

- 임시 DB 생성 명령 실행
- backup.sql 내용이 stdin으로 전달되는지
- cleanup=true일 때 DROP DATABASE가 실행되는지
- restore 실패 시 cleanup을 시도하는지
- unsafe database name을 거부하는지

Verifier 테스트에서는 다음을 검증했습니다.

- expectedTables가 모두 있으면 VERIFIED
- expected table이 없으면 FAILED
- verifyRowCount=false이면 Row Count를 생략
- expectedTables가 없으면 복원된 전체 테이블 기준으로 검증
- 복원된 테이블이 없으면 FAILED

Control Plane 테스트에서는 다음을 검증했습니다.

- 백업 Task 완료 후 Restore Verify Task 생성
- Restore Verify 성공 시 Job 성공
- Restore Verify 실패 시 Job 실패
- Cleanup 실패 시 Job 실패
- 복원 검증 결과와 Item 저장

## 설계상 남겨둔 부분

이번 작업에서 복원 검증 흐름은 연결했지만, 아직 개선할 부분도 남아 있습니다.

### 1. Backup Task Payload에 실제 DB 접속 정보 연결

현재 일부 백업 Task 생성 흐름은 아직 `databaseId` 중심으로 동작합니다.

하지만 Go Agent의 MySQL Backup Handler는 실제로 다음 값이 필요합니다.

```text
databaseName
host
port
username
password
```

따라서 이후에는 `ManagedDatabase`와 `DatabaseCredential`을 조회해서 백업 Task Payload에 접속 정보를 넣도록 개선해야 합니다.

---

### 2. 실패 Task의 resultPayload 저장

현재 AgentService 구조에서는 Handler가 error를 반환하면 Control Plane에 상세 resultPayload를 저장하지 않고 `FailTask`만 호출합니다.

즉, 실패 시에는 다음 정도만 전달됩니다.

```text
errorCode
errorMessage
```

복원 검증 실패의 상세 결과를 더 안정적으로 저장하려면, 실패 Task에도 resultPayloadJson을 저장할 수 있도록 API를 확장할 필요가 있습니다.

---

### 3. 대용량 DB 검증 전략

현재는 MVP 수준으로 `SHOW TABLES`와 `COUNT(*)`를 사용합니다.

하지만 대용량 테이블에서는 `COUNT(*)` 비용이 클 수 있습니다.

이후에는 다음 전략을 고려할 수 있습니다.

- Row Count 검증 옵션화
- 일부 핵심 테이블만 검증
- information_schema 기반 추정치 사용
- 샘플링 검증
- 테이블 checksum 검증
- restore timeout 정책 추가

---

## 정리

이번 작업에서는 백업 성공 기준을 단순 파일 생성에서 실제 복원 가능성 중심으로 확장했습니다.

핵심 변화는 다음과 같습니다.

```text
백업 파일 생성 성공
  ↓
복원 검증 Task 생성
  ↓
임시 DB에 restore
  ↓
테이블 검증
  ↓
결과 저장
  ↓
Job 성공/실패 판단
```

이 구조를 통해 DB FleetOps는 백업 파일이 존재한다는 사실뿐 아니라, 그 백업 파일이 실제로 복원 가능한지까지 관리할 수 있게 됐습니다.

운영 플랫폼 관점에서는 단순 자동화보다 중요한 것이 신뢰할 수 있는 자동화입니다.

이번 작업은 백업 자동화의 신뢰 기준을 높이기 위한 기반 작업이었습니다.
